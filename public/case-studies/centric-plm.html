Multi-agent GenAI PLM at Centric Software.

A multi-agent GenAI suite over Centric's PLM data — AI-assisted tech-pack generation, BOM intelligence, and design / material search — deployed to Tier-1 fashion brands including Asics and Under Armour.

The problem

PLM data is rich but unforgiving — styles, BOMs, suppliers, materials, sustainability and compliance attributes, all coupled to real-world manufacturing operations and audited regulatory regimes. Tier-1 fashion brands wanted natural-language workflows over this data without sacrificing the structural correctness PLM exists to enforce.

The brief: a GenAI PLM suite that's fast, multi-tenant, role-aware, and grounded enough to be trusted by sourcing and design teams shipping product to global markets.

Constraints

  • ·Multi-tenant from day one — strict isolation between brands sharing the same deployment.
  • ·Role-based access enforcement that survives every agent hop, not just the API gate.
  • ·Domain-specific accuracy: BOM correctness, material code mapping, sustainability and compliance flags must be ground-truthable.
  • ·Cost-per-query is a product-level KPI, not just an infra one — every release is gated against it.

Approach

01 · Multi-agent system

FastAPI + LangGraph orchestration over PLM entities. Specialised agents for tech-pack generation, BOM intelligence, and design / material search; shared retrieval and grounding layer; Bedrock Claude as the reasoning model.

02 · Tenant & role enforcement

Tenant context and RBAC are embedded in every retrieval path, not bolted on at the API layer. Agents can't see what their caller can't see, even mid-graph.

03 · PLM eval harness

Domain-specific eval covering BOM correctness, material code mapping, sustainability and compliance flags, groundedness, and cost-per-query. Every release across every customer rollout is gated against it.

04 · Workflow automation

Natural-language queries map to deterministic PLM workflows where possible — the LLM picks the path, the system runs the actual operation. Predictable side-effects, auditable history.

Reflections

PLM is a domain where "plausible" is a failure mode — a confidently-wrong material code breaks a real garment. The eval harness has to be domain-specific from day one; generic groundedness scores aren't enough.

The other lesson: lean on deterministic PLM operations whenever the LLM's job is "pick the right action." Generative reasoning is the navigation layer; the actual mutation should always be an audited, testable system call.
